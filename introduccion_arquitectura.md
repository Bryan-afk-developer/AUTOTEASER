### 3.1.2 Arquitectura del Sistema y Proceso de Desarrollo 

Una vez concluida la fase de levantamiento de requerimientos, se inició el diseño de la arquitectura de software que serviría como base para el desarrollo de la plataforma. Durante esta etapa se analizó la naturaleza del problema que debía resolver el sistema, identificando que el procesamiento documental involucraría archivos PDF de gran tamaño y formatos heterogéneos (incluyendo documentos escaneados), consultas continuas a modelos de Inteligencia Artificial, almacenamiento seguro en la nube y la necesidad de soportar a múltiples usuarios interactuando simultáneamente desde un portal web. 

Debido a la alta demanda de procesamiento de estas características, desde las primeras etapas del diseño se descartó una arquitectura monolítica en la que todas las responsabilidades se ejecutaran dentro de un único componente. En su lugar, se optó por implementar una arquitectura Cliente-Servidor fuertemente desacoplada, basada en servicios REST. Esta decisión separó por completo la capa de presentación visual (Frontend) de la lógica de negocio y procesamiento de datos (Backend), lo que permitió desarrollar ambos componentes de manera independiente y facilitó la integración escalonada de nuevos submódulos conforme maduraba el proyecto. 

En esta arquitectura, las interfaces web —divididas estratégicamente entre el Portal del Cliente y el Portal Administrativo— funcionan como consumidores asíncronos de la API principal, desarrollada en el framework FastAPI. La plataforma fue estructurada en torno a tres grandes pilares operativos, todos orquestados a través de solicitudes HTTP hacia el servidor:

1. **Expediente Rojo (Gestión Documental):** El flujo inicia cuando los clientes o analistas interactúan con este módulo para la carga de requerimientos. El frontend empaqueta los documentos y los envía al servidor, el cual se encarga de validar los archivos, sincronizarlos con la base de datos y la nube, y devolver indicadores de estado en tiempo real (Aprobado, Rechazado o En Revisión) para actualizar los semáforos visuales de la plataforma.
2. **AutoCAF (Consolidación Multianual):** Para el análisis financiero, el frontend envía peticiones que incluyen tanto los estados financieros históricos como las coordenadas visuales seleccionadas por el analista (recortes). El servidor intercepta estos datos, coordina la lectura mediante motores de Reconocimiento Óptico de Caracteres (OCR), delega la igualación contable al modelo de lenguaje (LLM) y ensambla dinámicamente un archivo de Microsoft Excel. Finalmente, el servidor retorna este archivo binario al navegador del analista para su descarga instantánea.
3. **AutoTeaser (Evaluación de Riesgo):** Como componente culminante, el sistema recibe la instrucción de generar el expediente maestro. El backend recolecta las extracciones previamente validadas de los otros módulos (como el *Score* del Buró de Crédito, la situación fiscal del SAT y la información legal de las Actas) y orquesta la inyección final de estos datos sobre la plantilla institucional de riesgo.

Bajo este modelo, una vez recibida cualquier petición, el backend actúa como el gran orquestador central: ejecuta estrictas reglas de negocio, interactúa transaccionalmente con la base de datos (Supabase), coordina las capas de Inteligencia Artificial y devuelve una respuesta estructurada al cliente. Esta respuesta puede abarcar desde una simple alerta visual en la pantalla, hasta la entrega de un documento financiero de Excel matemáticamente cuadrado y listo para el comité de crédito.
